GDPR Compliance in Big Data and Hadoop-based Systems by data minimization

Big data and Hadoop-based systems can offer efficient ways to manage and analyze large volumes of data at the cost of random access to records. The latter become essential particularly to comply with different data protection guidelines such as GDPR that often requires deleting specific “personal identifiable information” from the system. There are tools and storage that gives access to random records but it comes with its own challenge and forces the developer to carefully design each entity in the system individually.

Here are some of the key challenges:

  • Data deletion: Hadoop based systems are optimized for high-speed data ingestion and query processing.Identifying the lineage of a specific record and deleting it entirely from the system may be challenging.
  • Data retention: These systems may be optimized for storing large volumes of data for long periods of time, business decisions often benefit from longer data history as well. On the contrary organizations must ensure that they are only storing personal identifiable information for as long as necessary as per the data governance rules.
  • Data anonymization: Anonymizing data for a longer period for analytical use-case may require significant modifications to the database schema, queries and ETL, which can impact performance.
  • Data access controls: Various data protection policies require that organizations have appropriate data access controls in place to protect personal data. It can be challenging to implement fine-grained access controls.
In order to address these challenges, organizations may need to take a holistic approach to data management. This may involve implementing data retention policies, data access controls, and data encryption, as well as utilizing technologies that are specifically designed to support GDPR compliance in big data and Hadoop-based systems. With careful planning and the right tools, it's possible to design systems that comply with GDPR's requirements while still deriving useful insights from data. In this blog we will focus on such a data architecture with privacy compliance in its center, from the very beginning. The approach tries to put the information in an organized way that would help manage the personal identifiable information from a single place. It would also touch upon a few scopes for a more robust system but doesn’t go into the details. For the simplicity we would keep the discussion limited to a single area.

Create a Conformed User Dimension and Replacement Id mapping table

The first step in this approach is to create a conformed user dimension, the idea here is that this is the single truth user dimension that would be used across different downstream data marts. There will be a second table that would store an alternative replacement ID (RID) for each actual user_id in the conformed dimension table that will be used to anonymize the out of erasure period data. In the later section we will see how the combination of these 2 tables will help us to anonymize and unlink out of retention personal data while still preserving its usefulness for analytical purposes.

Redact User-specific Data

Next, any downstream table that holds user-specific data should only hold the actual user_ids up to the allowed erasure time period. Once the data is out of this period an anonymized version of the dataset would be created with the help of the RID mapping for the data. For Example let’s say the table data retention is 1 year, but the allowed erasure deadline is 90 days. So the actual table would hold the user_ids for 90 days and a separate table/older partitions would hold the RIDs instead of the user_ids. In order to generate the key of the conformed dimension a consistent hashing algorithm of the user email can be used. In that case, if the user rejoins with the same email, the user can be linked back with the old records. A separate process can run system wide that creates an anonymized version of the dataset, which replaces the user_id with the rid, and redacts the related columns. This process ensures that personal data is not linked back to an individual once the records are removed from the conformed user dimension but still can be used for numerical analysis.

Erase User Data

When a user requests their data to be deleted, the organization should nullify the mapping in the replacement Id table. This removes the link between the actual user, and there are no extra steps involved for downstream processes. Whereas the data with rid will still be useful and the data in the downstream will be eventually anonymized after the allowed erasure period of 90 days. This ensures that any data kept into the conformed dimension for any legal case can’t be linked to any business intelligence use cases. If a consistent hashing is used to generate the RID, it can still be used to link back only in legal cases. Finally, it's important for organizations to carefully review their systems and ensure that personal data is not inadvertently propagated beyond the allowed time period. The ETL process must not propagate the deleted user_id further, and the organization should keep track of any new data sources that could potentially introduce personal data.

Review Your Systems

Finally, it's important for organizations to carefully review their systems and ensure that personal data is not inadvertently propagated beyond the allowed time period. The ETL process must not propagate the deleted user_id further, and the organization should keep track of any new data sources that could potentially introduce personal data. In order to enable few functionalities outlined in this post it would be beneficial to store the table definition in code that would help marking the UII(User Identifiable information) fields. And the erasure process can use normal ETL, Machine learning and manual input to identify UII data. By following this approach, organizations can design their systems to comply with GDPR's requirements while still deriving useful insights from data. By using techniques such as data anonymization, retention policies, and aggregated fact tables, organizations can protect individuals' privacy while still benefiting from the power of big data and Hadoop-based systems. While there could be many nuances in data privacy, this article touches the tip of the iceberg. However, a simple process can be the most efficient one as it has fewer exceptions and lesser fault lines.
